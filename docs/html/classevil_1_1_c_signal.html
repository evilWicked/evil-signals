Detailed Description

Due to the complexities of managing interdependencies between classes that generate signals and the existence of classes that respond to signals as discussed in Design Ideas: Signals the signals and slots mechanism requires the following as a minimum

  1. A Signal Class (CSignal ) generates signals as well as maintains and manages a list of slots that the signal calls.
  2. The Signal class must inform any connected slot that it is being removed.
  3. The Signal class must remove any slot that it is notified of as being deleted.
  4. A Slot Class (CSlot) that provides the binding between the between the signal and the actual member function being called.
  5. The Slot class must inform the connected signal that it is being deleted.
  6. The Slot class must not attempt to communicate with a deleted signal.
  7. It is up to the object holding the slot (it may be a member) to remove any slots that are no longer receiving signals.

Thread Safety: In General Nope. A single signal on a single thread calling different slots on different threads is supported.

A single signal on multiple threads talking to the same slot or a single signal on a single thread talking to a single slot on multiple threads all at once, or a single signal on multiple threads talking to a single slot on multiple threads...... ROFL!!

Lets just say your mileage may vary.....

In the instance that you have different threads talking to a single signal then you can enable lock guards to ensure changes to the signals list of slots is thread safe by calling useLockGuards(true); It defaults to false.

See the discussion in Design Ideas: Signals
