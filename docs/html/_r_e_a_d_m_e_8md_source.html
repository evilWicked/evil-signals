8 Installation
9 ------------
10 evil signals is installed by simply the source similar to a header only implementation. The simplest way to do this is simply to copy the whole `evil-signals/` directory somewhere on your include path. Alternatively point your include path to wherever you have cloned the repo.
11 
12 include header files as usual using
13 
14  #include "evil/signals.h"
15  #include "evil/slots.h"
16 
17 To bring in the implementation files you can follow any of the approaches below.
18 
19 1. Either open a new .cpp file in your source somehere and simply include the implementations. You can open a new file for each .cpp file or bring them all into a single one. I use the latter.
20 
21  #include "evil/imp/signals.cpp"
22  #include "evil/imp/slots.cpp"
23 
24 2. Alternatively you can place the above implementation includes into an existing source file that contains your own code. If you take this approach be careful of namespaces. The evil classes reside inside the evil namespace and will have an identity crisis if you mix things up. Your system is likely to already have a signals.h file, so it may help to keep paths as `"evil/signals.h"` rather than `"signals.h"`
25 
26 3. If you want you can copy the source files out of evil/imp/ into your own code and use them like any other source.
27

51 Multithreading
52 --------------
53 
54 Multithreading. A can of worms...
55 
56 There is a long discussion about it in the documentation on the page Design Ideas: Signals which generally comes to the
57 conclusion that multithreading with signals and slots is not for the feint hearted. With that said evil-signals supports multi threading.
58 
59 To cover any situation where you have mutiple threads interacting with a single signal you need to enable lock guards on the signal.
60 
61  signal.useLockGuards(true);
62 
63 It defaults to false.
64 
65 To cover any situation where you have mutiple threads interacting with a single slot you need to enable lock guards on the slot.
66 
67  slot.useLockGuards(true);
68 
69 It also defaults to false.
70 
71 The fiollowing scenarios all fit under the umbrella of multithreading.
72 
73 ### Scenarios
74 1. A single signal on a single thread calling different slots on multiple threads.
75 2. A single signal on multiple threads talking to the same slot.
76 3. A single signal on a single thread talking to a single slot on multiple threads.
77 4. A single signal on multiple threads talking to a single slot on multiple threads.
78 5. Changing the state of a signal from several threads (not receiving a signal - which doesn't change state).
79 6. Changing the state a slot from several threads (not sending a signal - which doesn't change state).
80 
81 The only one I really recommend is the first. A single signal on a single thread calling different slots on multiple threads.
