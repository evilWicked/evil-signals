4 About
5 -----
6 
7 evil signals provides a simple lightweight C++ threadsafe signals and slots implementation. It contains both threadsafe and
8 non threadsafe versions. The normal (non threadsafe but faster) version consists of evil::Signal<...Args> paired with evil::Slot<...Args> .
9 The threadsafe pair evil::ThreadSignal<...Args> paired with evil::ThreadSlot<...Args> utilises a Read Write Lock that enables parallel dispatches together with sequential modifications.
10 
11 It is intended to be used as a source only distribution relying upon C++11 std libraries.

19 Installation
20 ------------
21 evil signals is installed by simply including the source. The important directory is the evil/ directory underneath evil-signals. This contains all dependencies and is compatible with the structure of other evil libraries allowing them to coexist. The simplest way to get things working is to copy (or move - not so good) the whole evil/ directory and all subdirectories from under `evil-signals/` to somewhere on your include path. Your include path needs to be the directory above evil/ so we can avoid confusion with other headers such as signal.h

68 Usage
69 -----
70 
71 Usage is fairly simple. Both the signal and the slot are templated to handle different arguments being passed between the signal
72 and the slot when a signal is dispatched.
73 
74  evil::Signal<...Args>
75  evil::Slot<...Args>
76 
77 As an example if we want to use callbacks of the form
78 
79  void foo(int i){...};
80 
81 ###Create a signal slot pair
82 
83 We can create a signal and slot that would enable an integer to be passed from the slot to the signal with:
84 
85  evil::Signal<int> sig;
86  evil::Slot<int> slot;
87 
88 ####Threadsafe
89 
90 The threadsafe versions are created using.
91 
92  evil::ThreadSignal<...Args> sig;
93  evil::ThreadSlot<...Args> slot;
94 
95 The only restriction is that callbacks need to have a void return. The signal has no way of
96 doing anything with a return value anyway.
97 
98 ###Attach a callback to the slot.
99 
100 To attach a callback to the slot you can use a standard function, a lambda or a class member.
101 Normal functions and lambdas use "attach" where classes that need an instance of the class to be bound
102 use "attach_class". So if foo is a function
103 
104  void foo(){}
105 
106 you would use;
107 
108  slot.attach(foo);
109 
110 
111 If C is a class with an instance c you would use
112 
113  class C{
114  public:
115  void foo(){}
116  }
117 
118  C c;
119 
120  slot.attach_class(&c,&C::foo);
121 
122 ###Connect the slot to the signal
123 
124 To connect the slot to the signal you use add from the signal.
125 
126  sig.add(&slot)
127 
128 similarly to disconnect them you use remove.
129 
130  sig.remove(&slot)
131 
132 The add signature allows an optional integer priority. Higher priority lots fire earlier. You
133 can also set a slot to fire once and automatically be removed. This can be useful when booting
134 into code where you are waiting for an event that will occur only once.
135 
136  void add(SlotBase *slot, int priority = 0, bool active = true);
137 
138 ###Dispatch a signal.
139 
140 To dispatch a signal you simply call dispatch passing any arguments to it, In the above
141 
142  sig.dispatch(1);
143 
144 would pass 1 to each attached slot when dispatch is called.
145 
146 ####Passing by reference.
147 
148 You can pass objects by reference to slots. This is not a constant reference, so be careful with this.
149 I had considered making it const but that would restrict what you do. In the choice between you being
150 careful or you not being allowed to do something - I take the Javascript view that its your code - do
151 what you want. This means that if you define
152 
153  evil::Signal<int&> sig;
154  evil::Slot<int&> slot1;
155  evil::Slot<int&> slot2;
156 
157  void foo(int &i){i++;}
158 
159  slot1.attach(foo);
160  slot2.attach(foo);
161 
162  sig.add(&slot1);
163  sig.add(&slot2);
164 
165  int i=0;
166  sig.dispatch(i);
167 
168 i would now be now equal to 2 because it would be incremented with every call of the slot by the signal.
169 
170 Slots can be deactivated and reactivated.
171 
172  slot.setActive(false);
173  slot.setActive(true);
174 
175 Multithreading
176 --------------
177 
178 There are two versions of the both the signals and the slots that are derived from the same underlying classes. The normal
179 versions evil::Signal and evil::Slot are not thread safe. The threadsafe classes are evil::ThreadSignal and evil::ThreadSlot
180 These are implemented using read write locks enabling parallel dispatch of signals alongside mutex locked changes to the signal
181 and slot.
